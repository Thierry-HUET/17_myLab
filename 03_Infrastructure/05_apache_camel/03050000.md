## Qu'est-ce qu'Apache Camel ?

Apache Camel est un **framework d'intégration open source** basé sur les *Enterprise Integration Patterns* (EIP), formalisés par Gregor Hohpe et Bobby Woolf. Il permet de connecter des systèmes hétérogènes — bases de données, files de messages, API REST, fichiers, protocoles réseau — via des **routes déclaratives**.

Lancé en 2007 sous l'égide de l'Apache Software Foundation, il dispose de plus de **300 composants** couvrant la quasi-totalité des protocoles et formats du marché : JDBC, JMS, Kafka, HTTP, FTP, SFTP, S3, CSV, JSON, XML, Avro, Parquet…

Les routes s'écrivent en Java DSL, YAML ou XML. Karavan fournit un éditeur visuel par-dessus ces formats.

## Camel vs ESB

Un ESB (*Enterprise Service Bus*) est une **plateforme complète** : il intègre routage, transformation, gouvernance, monitoring et registre de services. Camel n'est qu'un **framework de routage et transformation** — léger, embarquable, sans serveur dédié. Il peut néanmoins constituer le moteur d'un ESB (c'est le cas d'Apache ServiceMix).

:::{image} /static/img/camel/camel_vs_esb.svg
:alt: Camel comme sous-ensemble d'une plateforme ESB
:align: center
:width: 100%
:::

| Critère | Apache Camel | ESB commercial |
|---|---|---|
| Licence | Apache 2.0 — gratuit | Propriétaire ou freemium |
| Déploiement | Conteneur, microservice | Serveur dédié |
| Courbe d'apprentissage | Modérée (Java / YAML) | Élevée |
| Gouvernance | À construire | Incluse |
| Monitoring | Via Prometheus / Grafana | Intégré |
| Connecteurs propriétaires | Limités | Riches (SAP, Salesforce…) |
| Cible | Dev-driven, cloud, microservices | Équipes mixtes, legacy, enterprise |

:::{note}
Apache ServiceMix est un ESB open source construit sur Camel + ActiveMQ + Karaf.
:::

## Conditions d'usage

**Adapté quand**
- Les équipes sont à l'aise avec le code (Java ou YAML)
- L'architecture cible est microservices ou conteneurisée
- Le budget ne permet pas une licence ESB commerciale
- Les connecteurs standard couvrent les besoins (JDBC, REST, Kafka, FTP…)

**Moins adapté quand**
- Des utilisateurs métier non-développeurs maintiennent les flux
- Une gouvernance centralisée est requise d'emblée
- Des connecteurs propriétaires critiques sont nécessaires (SAP BAPI, IBM CICS…)

## Cas d'usage par type d'organisation

::::{grid} 1

:::{card} 🏪 PME — moins de 250 salariés
**Contexte typique** — ERP on-premise (Sage, Cegid), site e-commerce, CRM cloud.

- Synchronisation des commandes e-commerce → ERP via REST → JDBC
- Export comptable quotidien vers le cabinet : JDBC → CSV → SFTP
- Notification client à chaque changement de statut : JMS → SMTP

Zéro licence, déploiement sur un seul conteneur Docker, maintenance par un développeur junior.
:::

:::{card} 🏢 ETI — 250 à 5 000 salariés
**Contexte typique** — ERP Dynamics ou SAP B1, plusieurs filiales, BI Power BI ou Qlik, legacy MS-SQL.

- Réplication MS-SQL → PostgreSQL pour la BI *(cas traité dans ce module)*
- Agrégation multi-filiales : JDBC → Kafka → entrepôt central
- Alertes métier temps réel : Kafka → REST → Teams / Slack
- Interfaçage ERP ↔ WMS sans développement côté ERP

Découplage des systèmes, pagination sur grands volumes, déploiement route par route.
:::

:::{card} 🏛️ Grand groupe — plus de 5 000 salariés
**Contexte typique** — SAP S/4HANA, mainframe, Kafka enterprise, SI multi-pays.

- Moteur de routage dans une architecture event-driven (Camel + Kafka Streams)
- Remplacement progressif d'un ESB legacy par des microservices Camel indépendants
- Proxy de transformation entre zones de sécurité (DMZ → intranet)
- Intégration partenaires B2B (EDI, AS2, SFTP) sans middleware propriétaire

Granularité du déploiement, pas de *single point of failure*, compatible GitOps et CI/CD.

**Limite** — sans outillage complémentaire (Prometheus, Grafana, Jaeger), la visibilité sur les flux est insuffisante à cette échelle.
:::
