## Réseaux

Par défaut, chaque conteneur Docker est connecté à un réseau virtuel interne géré par le moteur. Ce réseau permet aux conteneurs de communiquer entre eux et avec l’extérieur, sans modifier la configuration réseau de la machine hôte.

Docker crée pour chaque réseau une interface virtuelle sur l’hôte (de type `bridge`) et attribue automatiquement une adresse IP à chaque conteneur connecté. Le trafic entre conteneurs transite par ce bridge ; le trafic vers l’extérieur passe par les règles NAT du système.

Trois **drivers** couvrent les cas d’usage courants :

- `bridge` (par défaut) : réseau privé isolé sur l’hôte, adapté à la grande majorité des déploiements
- `host` : le conteneur partage directement la pile réseau de l’hôte, sans isolation
- `none` : aucune interface réseau, isolation totale

La différence entre le réseau `bridge` créé par défaut et un réseau `bridge` personnalisé est importante en pratique : le réseau par défaut ne propose pas de résolution DNS par nom de conteneur, ce qui oblige à manipuler des adresses IP. Un réseau personnalisé active automatiquement un serveur DNS interne : les conteneurs se joignent par leur nom, ce qui est plus robuste et lisible.

Séparer les conteneurs sur des réseaux distincts permet également de cloisonner les flux : une base de données placée sur un réseau `backend` ne sera pas joignable depuis un conteneur qui n’est connecté qu’au réseau `frontend`. C’est le principe de moindre privilège appliqué au réseau.

### 01. Pile réseau Docker et drivers

::::{exercise}
:label: docker_ex10
:nonumber:

**Durée estimée : 30 min**

**Objectif** : comprendre la pile réseau Docker, inspecter les réseaux existants et observer le comportement des drivers standards.

**Étape 1 — Lister les réseaux existants**
```bash
docker network ls
docker network inspect bridge
```

**Étape 2 — Lancer deux conteneurs sur le réseau bridge par défaut**
```bash
docker run -d --name cont-a alpine sleep 3600  # -d detach, sleep maintient le conteneur actif
docker run -d --name cont-b alpine sleep 3600
```

Récupérer l'adresse IP de `cont-b` :
```bash
docker inspect cont-b --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}'
```

Tester la communication par IP (fonctionne) :
```bash
# Remplacer 172.17.0.X par l'IP obtenue ci-dessus
docker exec cont-a ping -c 3 172.17.0.X
```

Tester la communication par nom (échoue sur bridge par défaut) :
```bash
docker exec cont-a ping -c 2 cont-b
```

**Étape 4 — Nettoyer**
```bash
docker stop cont-a cont-b && docker rm cont-a cont-b
docker image prune -f
docker volume prune -f
```

| Driver | DNS par nom | Isolation | Usage |
|--------|-------------|-----------|-------|
| `bridge` (défaut) | Non | Partielle | Tests rapides |
| réseau personnalisé | Oui | Oui | Production |
| `host` | N/A | Aucune | Performance maximale |
| `none` | N/A | Totale | Sécurité maximale |

### 02. Réseaux personnalisés

::::{exercise}
:label: docker_ex11
:nonumber:

**Durée estimée : 45 min**

**Objectif** : créer un réseau bridge personnalisé, activer la résolution DNS inter-conteneurs et configurer des alias réseau.

**Étape 1 — Créer un réseau bridge personnalisé**
```bash
docker network create reseau-app    # driver bridge par defaut

docker network ls
docker network inspect reseau-app
```

**Étape 2 — Lancer des conteneurs sur ce réseau**
```bash
docker run -d \                 # mode detach
  --name serveur-web \          # nom du conteneur
  --network reseau-app \        # reseau personnalise a rejoindre
  nginx:alpine                   # image

docker run -d \                 # mode detach
  --name client \               # nom du conteneur
  --network reseau-app \        # meme reseau que serveur-web
  alpine \                      # image
  sleep 3600                     # maintient le conteneur actif
```

**Étape 3 — Tester la résolution DNS par nom**
```bash
docker exec client ping -c 3 serveur-web
docker exec client wget -qO- http://serveur-web | head -5
```

**Étape 4 — Configurer un alias réseau**
```bash
docker run -d \                 # mode detach
  --name serveur-web2 \         # nom du conteneur
  --network reseau-app \        # reseau a rejoindre
  --network-alias backend \     # alias DNS supplementaire sur ce reseau
  nginx:alpine                   # image

docker exec client ping -c 2 backend
```

**Étape 5 — Inspecter le réseau avec les conteneurs connectés**
```bash
docker network inspect reseau-app
```

**Étape 6 — Nettoyer**
```bash
docker stop serveur-web client serveur-web2
docker rm serveur-web client serveur-web2
docker network rm reseau-app
docker image prune -f
docker volume prune -f
```

:::{note}
Un alias réseau (`--network-alias`) permet à plusieurs conteneurs de répondre au même nom DNS — utile pour simuler un load balancer ou regrouper des services sous un nom fonctionnel.

**Exercice autonomie A02 — WordPress avec réseau personnalisé**

**Durée estimée : 30 min**

L'objectif est de déployer WordPress et MariaDB sur un réseau personnalisé, de façon à ce que WordPress puisse joindre la base de données par son nom de conteneur.

Utiliser un réseau personnalisé (plutôt que le bridge par défaut) est indispensable ici : c'est lui qui active la résolution DNS par nom, permettant à WordPress de trouver MariaDB sans connaître son adresse IP.

**Étape 1** — Créer un réseau bridge personnalisé nommé `wp-net`. Vérifier sa création.

**Étape 2** — Lancer un conteneur MariaDB 10.11 nommé `wp-db` sur le réseau `wp-net`. Configurer la base `wordpress`, l'utilisateur `wp` et les mots de passe via les variables d'environnement de l'image. Attendre que le log indique que MariaDB est prêt à accepter des connexions.

**Étape 3** — Lancer un conteneur WordPress nommé `wp-app` sur le même réseau `wp-net`, en exposant le port 8080. Connecter WordPress à MariaDB par son nom de conteneur via les variables d'environnement de l'image.

**Étape 4** — Vérifier que la page d'installation WordPress s'affiche sur `http://localhost:8080`. Depuis le conteneur `wp-app`, tester que `wp-db` est joignable par son nom.

**Étape 5** — Nettoyer : arrêter et supprimer les conteneurs, puis supprimer le réseau.

**Critères de validation**

- La page d'installation WordPress s'affiche sur `http://localhost:8080`
- `wp-db` est joignable par nom depuis `wp-app`
- Après nettoyage, `docker network ls` ne liste plus `wp-net`
::::

::::{solution} docker_ex11_auto
:label: sol_docker_ex11_auto
:class: dropdown

**Étape 1 — Créer le réseau**
```bash
docker network create wp-net    # reseau bridge personnalise
docker network ls
```

**Étape 2 — Lancer MariaDB**
```bash
docker run -d \                        # mode detach
  --name wp-db \                       # nom du conteneur (utilise comme nom DNS)
  --network wp-net \                   # reseau personnalise
  -e MYSQL_ROOT_PASSWORD=rootpass \    # mot de passe root
  -e MYSQL_DATABASE=wordpress \        # base a creer au demarrage
  -e MYSQL_USER=wp \                   # utilisateur applicatif
  -e MYSQL_PASSWORD=wppass \           # mot de passe utilisateur
  mariadb:10.11                        # image

docker logs wp-db   # attendre la ligne "ready for connections"
```

**Étape 3 — Lancer WordPress**
```bash
docker run -d \                              # mode detach
  --name wp-app \                            # nom du conteneur
  --network wp-net \                         # meme reseau que wp-db
  -p 8080:80 \                               # port-hote:port-conteneur
  -e WORDPRESS_DB_HOST=wp-db \              # nom DNS du conteneur MariaDB
  -e WORDPRESS_DB_USER=wp \                 # utilisateur de la base
  -e WORDPRESS_DB_PASSWORD=wppass \         # mot de passe
  -e WORDPRESS_DB_NAME=wordpress \          # nom de la base
  wordpress:latest                           # image
```

**Étape 4 — Vérifier**
```bash
# Ouvrir http://localhost:8080 dans le navigateur

# Tester la resolution DNS depuis wp-app
docker exec wp-app ping -c 2 wp-db
```

**Résultat attendu du ping :**
```
PING wp-db (192.168.x.2): 56 data bytes
64 bytes from 192.168.x.2: seq=0 ttl=64 time=0.091 ms
```

**Étape 5 — Nettoyer**
```bash
docker stop wp-app wp-db
docker rm wp-app wp-db
docker network rm wp-net
docker image prune -f
docker volume prune -f
```

:::{note}
`WORDPRESS_DB_HOST=wp-db` fonctionne grâce au réseau personnalisé : Docker y active un serveur DNS interne qui résout les noms de conteneurs. Sur le réseau bridge par défaut, ce mécanisme n'existe pas.
:::

### 03. Isolation réseau entre piles (Namespace net)

::::{exercise}
:label: docker_ex12
:nonumber:

**Durée estimée : 30 min**

**Objectif** : démontrer que deux piles applicatives sur des réseaux distincts sont invisibles l'une de l'autre — isolation garantie par le **Namespace net** assigné à chaque conteneur.

Chaque réseau Docker personnalisé correspond à un Namespace net isolé. Un conteneur connecté au réseau A ne voit pas les conteneurs du réseau B, même s'ils s'exécutent sur le même hôte physique.

**Étape 1 — Créer deux réseaux distincts**
```bash
docker network create pile-frontend
docker network create pile-backend
```

**Étape 2 — Déployer des conteneurs sur chaque réseau**
```bash
docker run -d --name web-1 --network pile-frontend nginx:alpine  # reseau frontend
docker run -d --name web-2 --network pile-frontend nginx:alpine  # meme reseau
docker run -d --name db-1  --network pile-backend  alpine sleep 3600  # reseau backend isole
```

**Étape 3 — Vérifier la communication intra-réseau**
```bash
# web-1 atteint web-2 (même réseau)
docker exec web-1 ping -c 2 web-2
```

**Étape 4 — Vérifier l'isolation inter-réseaux**
```bash
# web-1 ne voit pas db-1 (réseaux différents)
docker exec web-1 ping -c 2 db-1
```

**Étape 5 — Connecter un conteneur aux deux réseaux (passerelle contrôlée)**
```bash
docker run -d \                 # mode detach
  --name proxy \                # nom du conteneur
  --network pile-frontend \     # reseau initial
  nginx:alpine                   # image

docker network connect pile-backend proxy

# Le proxy atteint maintenant db-1
docker exec proxy ping -c 2 db-1
# web-1 ne peut toujours pas atteindre db-1 directement
docker exec web-1 ping -c 2 db-1
```

**Étape 6 — Vérifier l'appartenance aux réseaux**
```bash
docker network inspect pile-frontend --format '{{range .Containers}}{{.Name}} {{end}}'
docker network inspect pile-backend  --format '{{range .Containers}}{{.Name}} {{end}}'
```

**Étape 7 — Nettoyer**
```bash
docker stop web-1 web-2 db-1 proxy
docker rm web-1 web-2 db-1 proxy
docker network rm pile-frontend pile-backend
docker image prune -f
docker volume prune -f
```
::::

::::{solution} docker_ex12
:label: sol_docker_ex12
:class: dropdown

**Étape 3 — ping intra-réseau (succès attendu) :**
```
PING web-2 (192.168.x.3): 56 data bytes
64 bytes from 192.168.x.3: seq=0 ttl=64 time=0.091 ms
```

**Étape 4 — ping inter-réseaux (isolation attendue) :**
```

```

db-1 est inaccessible depuis web-1. Ils partagent le même hôte physique mais leurs **Namespaces net** sont distincts — aucune route ne les relie.

**Étape 5 — après `docker network connect` :**

Le conteneur `proxy` dispose de deux interfaces réseau (une par réseau) et peut joindre les deux piles. `web-1` reste isolé de `db-1`.

**Étape 6 — Résultat attendu :**
```
pile-frontend : web-1 web-2 proxy
pile-backend  : db-1 proxy
```

`web-1` n'apparaît que dans `pile-frontend` : il ne peut pas joindre `db-1`. `proxy`, connecté aux deux réseaux, apparaît dans les deux listes.

| Situation | Résultat |
|-----------|----------|
| Conteneurs sur le même réseau | Communication par nom DNS |
| Conteneurs sur réseaux différents | Isolation totale |
| Conteneur connecté aux deux réseaux | Passerelle contrôlée |

**Exercice autonomie A03 — WordPress avec deux réseaux séparés**

**Durée estimée : 30 min**

L'objectif est de déployer WordPress et MariaDB sur deux réseaux distincts, en connectant WordPress aux deux pour qu'il puisse joindre la base tout en restant isolé du reste. Cette architecture sépare le trafic web (frontend) du trafic base de données (backend).

Séparer les réseaux est une bonne pratique de sécurité : MariaDB n'est pas directement accessible depuis le réseau frontend. Seul WordPress, connecté aux deux réseaux, peut joindre la base de données.

**Étape 1** — Créer deux réseaux : `wp-frontend` (Nginx et WordPress) et `wp-backend` (WordPress et MariaDB). Vérifier leur création.

**Étape 2** — Lancer MariaDB nommé `wp-db` sur le réseau `wp-backend` uniquement. Utiliser les mêmes variables d'environnement que l'exercice 02.

**Étape 3** — Lancer WordPress nommé `wp-app` sur le réseau `wp-frontend`, puis le connecter également au réseau `wp-backend`. Exposer le port 8080.

**Étape 4** — Vérifier que la page d'installation WordPress s'affiche sur `http://localhost:8080`. Vérifier que `wp-db` est joignable depuis `wp-app`. Confirmer via `docker network inspect` que `wp-db` n'apparaît que dans `wp-backend`.

**Étape 5** — Nettoyer : arrêter et supprimer les conteneurs, puis supprimer les deux réseaux.

**Critères de validation**

- La page d'installation WordPress s'affiche sur `http://localhost:8080`
- `wp-db` est joignable par nom depuis `wp-app`
- `docker network inspect wp-backend` liste `wp-db`, `docker network inspect wp-frontend` ne le liste pas
- Après nettoyage, `docker network ls` ne liste plus `wp-frontend` ni `wp-backend`
::::

::::{solution} docker_ex12_auto
:label: sol_docker_ex12_auto
:class: dropdown

**Étape 1 — Créer les réseaux**
```bash
docker network create wp-frontend   # reseau web
docker network create wp-backend    # reseau base de donnees
docker network ls
```

**Étape 2 — Lancer MariaDB sur le backend uniquement**
```bash
docker run -d \                        # mode detach
  --name wp-db \                       # nom du conteneur
  --network wp-backend \               # reseau backend uniquement
  -e MYSQL_ROOT_PASSWORD=rootpass \    # mot de passe root
  -e MYSQL_DATABASE=wordpress \        # base a creer
  -e MYSQL_USER=wp \                   # utilisateur applicatif
  -e MYSQL_PASSWORD=wppass \           # mot de passe
  mariadb:10.11

docker logs wp-db   # attendre "ready for connections"
```

**Étape 3 — Lancer WordPress et le connecter aux deux réseaux**
```bash
docker run -d \                              # mode detach
  --name wp-app \                            # nom du conteneur
  --network wp-frontend \                    # reseau frontend au lancement
  -p 8080:80 \                               # port-hote:port-conteneur
  -e WORDPRESS_DB_HOST=wp-db \             # nom DNS de MariaDB
  -e WORDPRESS_DB_USER=wp \                # utilisateur
  -e WORDPRESS_DB_PASSWORD=wppass \        # mot de passe
  -e WORDPRESS_DB_NAME=wordpress \         # nom de la base
  wordpress:latest

# Connecter wp-app au reseau backend pour joindre MariaDB
docker network connect wp-backend wp-app
```

**Étape 4 — Vérifier**
```bash
# WordPress joignable depuis le navigateur : http://localhost:8080

# wp-db joignable depuis wp-app (deux reseaux)
docker exec wp-app ping -c 2 wp-db
```

**Résultat attendu du ping depuis wp-app :**
```
PING wp-db (172.18.0.2): 56 data bytes
64 bytes from 172.18.0.2: seq=0 ttl=64 time=0.091 ms
```

**Étape 5 — Nettoyer**
```bash
docker stop wp-app wp-db
docker rm wp-app wp-db
docker network rm wp-frontend wp-backend
docker image prune -f
docker volume prune -f
```

**Architecture des réseaux :**
```
[wp-frontend]          [wp-backend]
   wp-app    <-------> wp-app <-------> wp-db
   (port 8080)         (pont entre
                        les deux reseaux)
```

WordPress joue le rôle de passerelle contrôlée : seul lui peut joindre MariaDB. Le trafic web (port 8080) reste sur le réseau frontend.

### 04. Bind mount

::::{exercise}
:label: docker_ex09a
:nonumber:

**Durée estimée : 20 min**

**Objectif** : utiliser un bind mount pour partager un répertoire hôte dans un conteneur et observer la propagation immédiate des modifications.

**Étape 1 — Préparer le contenu local**
```bash
mkdir -p /tmp/site-web
echo "<h1>Version initiale</h1>" > /tmp/site-web/index.html
```

**Étape 2 — Lancer Nginx avec bind mount**
```bash
docker run -d \                              # mode detach : tourne en arriere-plan
  --name nginx-bind \                        # nom du conteneur
  -p 8080:80 \                               # port-hote:port-conteneur
  -v /tmp/site-web:/usr/local/apache3/htdocs:ro \ # chemin-hote:chemin-conteneur:ro (lecture seule)
  nginx:alpine                                # image a utiliser

curl http://localhost:8080
```

**Étape 3 — Modifier le contenu depuis l'hôte sans redémarrer le conteneur**
```bash
echo "<h1>Modified from host</h1>" > /tmp/site-web/index.html
curl http://localhost:8080
```

**Étape 4 — Nettoyer**
```bash
docker stop nginx-bind && docker rm nginx-bind
rm -rf /tmp/site-web
docker image prune -fa
docker volume prune -fa
```
::::

::::{solution} docker_ex09a
:label: sol_docker_ex09a
:class: dropdown

**Résultat attendu après modification :**
```
<h1>Modified from host</h1>
```

La modification est immédiate — pas de redémarrage du conteneur nécessaire. C'est le comportement attendu d'un bind mount : le système de fichiers hôte est monté directement dans le conteneur via le **Namespace mnt**.

:::{note}
**Bind mount vs Volume nommé** : le bind mount dépend de la structure du système hôte — utile en développement. Le volume nommé est géré par Docker, portable, et recommandé en production.
:::

### 05. Volumes de données

::::{exercise}
:label: docker_ex08
:nonumber:

**Durée estimée : 45 min**

**Objectif** : maîtriser la création de volumes, leur audit et les opérations de sauvegarde, migration et restauration.

**Étape 1 — Créer un volume nommé**
```bash
docker volume create data-app
docker volume ls
docker volume inspect data-app
```

**Étape 2 — Écrire des données dans le volume**
```bash
docker run --rm \              # supprime le conteneur apres execution
  -v data-app:/data \          # volume-nomme:point-de-montage
  alpine \                     # image
  sh -c "echo 'Initial entry - $(date)' > /data/journal.txt && cat /data/journal.txt"
```

**Étape 3 — Vérifier la persistance sur un second conteneur**
```bash
docker run --rm \              # supprime le conteneur apres execution
  -v data-app:/data \          # meme volume : les donnees sont persistees
  alpine \                     # image
  cat /data/journal.txt
```

**Étape 4 — Audit du volume** (Linux uniquement)
```bash
docker volume inspect data-app --format '{{.Mountpoint}}'
sudo ls -lh $(docker volume inspect data-app --format '{{.Mountpoint}}')
```

**Étape 5 — Sauvegarde du volume**
```bash
docker run --rm \              # supprime le conteneur apres execution
  -v data-app:/data \          # volume source a sauvegarder
  -v /tmp:/backup \            # repertoire hote pour stocker l archive
  alpine \                     # image
  tar czf /backup/backup-data-app.tar.gz -C /data .
ls -lh /tmp/backup-data-app.tar.gz
```

**Étape 6 — Restauration sur un nouveau volume**
```bash
docker volume create data-app-restored

docker run --rm \              # supprime le conteneur apres execution
  -v data-app-restored:/data \ # volume de destination
  -v /tmp:/backup \            # repertoire hote contenant l archive
  alpine \                     # image
  tar xzf /backup/backup-data-app.tar.gz -C /data

docker run --rm \              # supprime le conteneur apres execution
  -v data-app-restored:/data \ # volume restaure
  alpine \                     # image
  cat /data/journal.txt
```

**Étape 7 — Nettoyer**
```bash
docker volume rm data-app data-app-restored
rm /tmp/backup-data-app.tar.gz
docker image prune -fa
docker volume prune -fa
```

Créer le fichier de configuration Nginx :
```bash
mkdir -p /tmp/nginx-lb
```

Contenu de `/tmp/nginx-lb/nginx.conf` :
```nginx
events {
    worker_connections 1024;
}

http {
    # Declaration du groupe de serveurs upstream (load balancing)
    upstream wordpress_pool {
        server wp-app-1:80;   # instance WordPress 1
        # server wp-app-2:80; # decommenter pour le load balancing
    }

    server {
        listen 80;

        location / {
            # Transmettre les requetes a l upstream WordPress
            proxy_pass         http://wordpress_pool;

            # En-tetes transmis a WordPress pour qu il connaisse l IP du client
            proxy_set_header   Host              $host;
            proxy_set_header   X-Real-IP         $remote_addr;
            proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
            proxy_set_header   X-Forwarded-Proto $scheme;
        }
    }
}
```

Lancer Nginx :
```bash
docker run -d \                                    # mode detach
  --name nginx-lb \                                # nom du conteneur
  --network wp-frontend \                          # reseau frontend
  -p 8080:80 \                                     # seul port expose sur l hote
  -v /tmp/nginx-lb/nginx.conf:/etc/nginx/nginx.conf:ro \ # configuration en bind mount
  nginx:alpine                                     # image legere
```

**Étape 6 (avancé) — Load balancing sur deux WordPress**
```bash
# Lancer un second WordPress
docker run -d \                              # mode detach
  --name wp-app-2 \                          # deuxieme instance
  --network wp-frontend \                    # reseau frontend
  -v wp-logs:/var/log/apache2 \             # meme volume de logs
  -e WORDPRESS_DB_HOST=wp-db \             # meme base de donnees
  -e WORDPRESS_DB_USER=wp \
  -e WORDPRESS_DB_PASSWORD=wppass \
  -e WORDPRESS_DB_NAME=wordpress \
  wordpress:latest

docker network connect wp-backend wp-app-2

# Mettre a jour nginx.conf : decommenter la ligne wp-app-2
# Puis recharger Nginx sans interruption
docker exec nginx-lb nginx -s reload

# Generer du trafic et observer la distribution
for i in $(seq 1 6); do curl -s http://localhost:8080 > /dev/null; done
docker logs wp-app-1
docker logs wp-app-2
```

**Rôle des directives Nginx :**

| Directive | Rôle |
|-----------|-------|
| `upstream wordpress_pool` | Définit le groupe de serveurs en amont |
| `server wp-app-1:80` | Ajoute une instance au pool (nom DNS Docker) |
| `proxy_pass http://wordpress_pool` | Envoie les requêtes vers le pool |
| `proxy_set_header Host` | Transmet le nom de domaine à WordPress |
| `proxy_set_header X-Real-IP` | Transmet l'IP réelle du client |
| `proxy_set_header X-Forwarded-For` | Historique des proxies traversés |

Par défaut, Nginx utilise **round-robin** : les requêtes sont distribuées à tour de rôle entre les instances du pool. D'autres algorithmes sont disponibles : `least_conn` (instance la moins chargée), `ip_hash` (affinité de session par IP client).

```bash
docker image prune -fa
docker volume prune -fa
```

### 07. Volume tmpfs — données sensibles en mémoire (Namespace mnt)

::::{exercise}
:label: docker_ex13
:nonumber:

**Durée estimée : 20 min**

**Objectif** : monter un volume tmpfs pour stocker des données sensibles uniquement en RAM, sans écriture sur le disque — illustre le **Namespace mnt** qui donne à chaque conteneur sa propre vue du système de fichiers.

Le Namespace mnt isole l'arborescence de fichiers du conteneur. Docker en tire parti pour proposer différents types de montages : volume nommé (géré par Docker), bind mount (répertoire hôte), et **tmpfs** (RAM uniquement, jamais persisté).

**Étape 1 — Lancer un conteneur avec un montage tmpfs**
```bash
docker run -d \                                        # mode detach
  --name app-secrets \                                 # nom du conteneur
  --tmpfs /run/secrets:rw,noexec,nosuid,size=10m \    # montage tmpfs : chemin:options
  alpine \                                             # image
  sleep 3600                                            # maintient le conteneur actif
```

**Étape 2 — Écrire un secret en mémoire**
```bash
docker exec app-secrets sh -c "echo 'temp-password' > /run/secrets/db_password"
docker exec app-secrets cat /run/secrets/db_password
```

**Étape 3 — Vérifier l'absence de trace sur le disque hôte** (Linux uniquement)
```bash
PID=$(docker inspect --format '{{.State.Pid}}' app-secrets)
# Le montage tmpfs est visible dans /proc mais absent du système de fichiers hôte
grep tmpfs /proc/$PID/mounts
sudo ls /var/lib/docker/volumes/  # le secret n'y figure pas
```

**Étape 4 — Confirmer la disparition à l'arrêt**
```bash
docker stop app-secrets && docker start app-secrets
docker exec app-secrets cat /run/secrets/db_password
# Fichier absent : le tmpfs a été réinitialisé
```

**Étape 5 — Comparer avec un volume nommé classique**
```bash
docker run -d \                 # mode detach
  --name app-volume \           # nom du conteneur
  -v secrets-vol:/run/secrets \ # volume nomme : persiste sur disque (contrairement a tmpfs)
  alpine \                      # image
  sleep 3600                     # maintient le conteneur actif

docker exec app-volume sh -c "echo 'persisted-password' > /run/secrets/db_password"
docker stop app-volume && docker start app-volume
docker exec app-volume cat /run/secrets/db_password
# La donnée est toujours présente : le volume nommé est persisté sur disque
```

**Étape 6 — Nettoyer**
```bash
docker stop app-secrets app-volume
docker rm app-secrets app-volume
docker volume rm secrets-vol
docker image prune -fa
docker volume prune -fa
```

```
persisted-password
```

| Type de montage | Persisté sur disque | Survit au redémarrage | Usage typique |

| `tmpfs` | Non (RAM) | Non | Secrets, tokens, cache temporaire |
| Volume nommé | Oui | Oui | Bases de données, uploads |

### 08. Partage de Namespace entre conteneurs — pattern sidecar

::::{exercise}
:label: docker_ex14
:nonumber:

**Durée estimée : 30 min**

**Objectif** : partager les Namespaces PID et réseau entre deux conteneurs pour implémenter un pattern sidecar — technique héritée de LXC où plusieurs processus cohabitaient dans un même espace d'isolation.

Sous LXC, un conteneur pouvait regrouper plusieurs processus partageant le même Namespace. Docker va plus loin en permettant de lier deux conteneurs distincts sur un ou plusieurs Namespaces spécifiques. C'est le fondement du pattern **sidecar** utilisé dans Kubernetes.

**Partie A — Partage du Namespace réseau**

**A1 — Lancer le conteneur principal**
```bash
docker run -d \                 # mode detach
  --name app-principale \       # nom du conteneur principal
  nginx:alpine                   # image
```

**A2 — Lancer un sidecar partageant le même Namespace réseau**
```bash
docker run -d \                              # mode detach
  --name sidecar-reseau \                    # nom du sidecar
  --network container:app-principale \       # partage le Namespace net du conteneur cible
  alpine \                                   # image
  sleep 3600                                  # maintient le conteneur actif
```

**A3 — Vérifier que les deux conteneurs partagent la même interface réseau**
```bash
docker exec app-principale ip addr show
docker exec sidecar-reseau ip addr show
# Les adresses IP sont identiques
```

**A4 — Le sidecar peut interroger nginx via localhost**
```bash
docker exec sidecar-reseau wget -qO- http://localhost:80 | head -3
```

---

**Partie B — Partage du Namespace PID**

**B1 — Lancer un conteneur principal**
```bash
docker run -d \                 # mode detach
  --name app-pid \              # nom du conteneur principal
  nginx:alpine                   # image
```

**B2 — Lancer un sidecar partageant le Namespace PID**
```bash
docker run -d \                   # mode detach
  --name sidecar-pid \            # nom du sidecar
  --pid container:app-pid \       # partage le Namespace PID du conteneur cible
  alpine \                        # image
  sleep 3600                       # maintient le conteneur actif
```

**B3 — Le sidecar voit les processus du conteneur principal**
```bash
docker exec sidecar-pid ps aux
# Les processus nginx apparaissent
```

**B4 — Vérifier depuis l'hôte** (Linux uniquement)
```bash
PID_APP=$(docker inspect --format '{{.State.Pid}}' app-pid)
PID_SIDE=$(docker inspect --format '{{.State.Pid}}' sidecar-pid)

ls -la /proc/$PID_APP/ns/pid
ls -la /proc/$PID_SIDE/ns/pid
# Inodes différents : les Namespaces PID sont partagés mais les conteneurs restent distincts
```

**Nettoyer**
```bash
docker stop app-principale sidecar-reseau app-pid sidecar-pid
docker rm app-principale sidecar-reseau app-pid sidecar-pid
docker image prune -fa
docker volume prune -fa
```

### 09. Conteneurs rootless — héritage LXC et User Namespace

::::{exercise}
:label: docker_ex15
:nonumber:

**Durée estimée : 30 min** — Linux uniquement

**Objectif** : comprendre le **User Namespace** et exécuter Docker en mode rootless — évolution directe du modèle LXC unprivileged containers introduit par Stéphane Graber en 2013.

Sous LXC classique, lancer un conteneur nécessitait les droits root sur l'hôte. Les **LXC unprivileged containers** ont introduit le User Namespace : un processus peut être `root` (UID 0) à l'intérieur du conteneur tout en étant un utilisateur sans privilège sur l'hôte. Docker rootless reprend ce mécanisme.

**Étape 1 — Observer le User Namespace par défaut**
```bash
# L'utilisateur à l'intérieur du conteneur
docker run --rm alpine id
# uid=0(root) gid=0(root)

# Le même processus vu depuis l'hôte
docker run -d --name test-uid alpine sleep 3600  # -d detach, sleep maintient le conteneur actif
PID=$(docker inspect --format '{{.State.Pid}}' test-uid)
ps -o pid,uid,user,comm -p $PID
# L'UID hôte est 0 (root) — mode privilégié par défaut
```

**Étape 2 — Lancer un conteneur avec un utilisateur non-root**
```bash
docker run --rm --user 1000:1000 alpine id  # --user uid:gid : utilisateur non-root
# uid=1000 gid=1000

# Verifier que l'ecriture dans / est refusee
docker run --rm \               # supprime le conteneur apres execution
  --user 1000:1000 \            # force l execution en tant qu utilisateur non-root
  alpine \                      # image
  sh -c "touch /test" 2>&1      # redirige stderr vers stdout pour voir l erreur
```

**Étape 3 — Inspecter le User Namespace**
```bash
ls -la /proc/$PID/ns/user
# Comparer avec le Namespace user de l'hôte
ls -la /proc/1/ns/user
```

**Étape 4 — Lire le mapping UID** (Linux uniquement)
```bash
cat /proc/$PID/uid_map
# Format : UID_container  UID_host  range
# En mode standard : 0  0  4294967295 (pas de remappage)
```

**Étape 5 — Activer le remappage d'UID (userns-remap)**

:::{note}
Cette étape modifie la configuration du daemon Docker. Elle est optionnelle et nécessite de redémarrer Docker.
:::

```bash
# Ajouter la configuration dans /etc/docker/daemon.json
sudo bash -c 'cat > /etc/docker/daemon.json <<EOF
{
  "userns-remap": "default"
}
EOF'

sudo systemctl restart docker

# Lancer un conteneur et observer le remappage
docker run -d --name test-remap alpine sleep 3600  # -d detach, sleep maintient le conteneur actif
PID_REMAP=$(docker inspect --format '{{.State.Pid}}' test-remap)
cat /proc/$PID_REMAP/uid_map
# UID 0 dans le conteneur est mappé à un UID élevé sur l'hôte
```

**Étape 6 — Restaurer la configuration par défaut**
```bash
sudo bash -c 'echo "{}" > /etc/docker/daemon.json'
sudo systemctl restart docker
```

**Étape 7 — Nettoyer**
```bash
docker stop test-uid 2>/dev/null; docker rm test-uid 2>/dev/null
docker stop test-remap 2>/dev/null; docker rm test-remap 2>/dev/null
docker image prune -fa
docker volume prune -fa
```

**Étape 1 — id dans le conteneur :**
```
uid=0(root) gid=0(root) groups=0(root)
```

**Étape 1 — ps depuis l'hôte :**
```
PID   UID  USER   COMM
3842    0  root   sleep
```

En mode standard, le root du conteneur est le root de l'hôte. Si un processus s'échappe du conteneur, il a les droits root sur la machine.

**Étape 2 — avec --user 1000 :**
```
uid=1000 gid=1000
touch: /test: Permission denied
```

**Étape 4 — uid_map sans remappage :**
```
0  0  4294967295
```

Mapping identité : UID 0 dans le conteneur = UID 0 sur l'hôte.

**Étape 5 — uid_map avec userns-remap :**
```
0  100000  65536
```

UID 0 dans le conteneur est mappé à l'UID 100000 sur l'hôte. Un processus échappé n'a aucun privilège sur la machine.

| Mode | UID dans le conteneur | UID sur l'hôte | Risque |
|------|-----------------------|----------------|--------|
| Standard | 0 (root) | 0 (root) | Élevé |
| `--user 1000` | 1000 | 1000 | Faible |
| `userns-remap` | 0 (root apparent) | 100000+ | Faible |
| Docker rootless | 0 (root apparent) | UID courant | Minimal |

:::{note}
**Filiation LXC → Docker** : les LXC unprivileged containers (2013) ont démontré la viabilité du User Namespace pour l'isolation sans privilèges. Docker a intégré `userns-remap` en 2016 (v1.10), puis le mode rootless complet en 2019 (v19.03). Aujourd'hui, Docker rootless est l'approche recommandée pour les environnements multi-utilisateurs et les pipelines CI/CD.
