## Prérequis

- Connaissance de Linux (ligne de commande, arborescence, droits)
- Notions de réseau TCP/IP (adresses IP, ports, protocoles)
- Accès à un terminal avec droits administrateur (sudo)
- Poste sous Linux, macOS ou Windows

:::{note}
**Windows** : tous les exercices supposent l'utilisation de **WSL2** (Windows Subsystem for Linux). Docker Desktop sur Windows utilise WSL2 comme backend — il est donc déjà présent. Ouvrir un terminal WSL2 (Ubuntu) plutôt que PowerShell pour exécuter les commandes.

Installation WSL2 si absent :
```powershell
wsl --install
```
Puis redémarrer et ouvrir **Ubuntu** depuis le menu démarrer.
:::

## Installation et architecture

### 01. Installation de Docker

::::{exercise}
:label: docker_ex01
:nonumber:

**Durée estimée : 30 min**

Toutes les commandes sont à saisir dans un terminal (console) avec les droits suffisants.

Installer Docker et vérifier que le moteur fonctionne correctement.

**Étape 1 — Ouvrir un terminal**

- Linux : terminal natif (Bash, Zsh)
- macOS : Terminal ou iTerm2
- Windows : PowerShell ou Windows Terminal (WSL2 requis)

**Étape 2 — Installation**

Linux (Ubuntu/Debian) :

```{code-cell} bash
sudo apt-get update
sudo apt-get install ca-certificates curl gnupg
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo apt-get install docker-ce docker-ce-cli containerd.io
```

Windows / macOS : télécharger Docker Desktop depuis [https://www.docker.com/products/docker-desktop](https://www.docker.com/products/docker-desktop)

**Étape 3 — Vérification**

```{code-cell} bash
docker --version
docker info
```

**Étape 4 — Premier conteneur**

```{code-cell} bash
docker run hello-world
```

**Étape 5 — Explorer l'arborescence Docker** (Linux uniquement)

```{code-cell} bash
sudo ls /var/lib/docker/
```

**Critères de validation**

- `docker --version` retourne un numéro de version sans erreur
- `docker run hello-world` affiche le message de bienvenue
- `docker info` affiche l'état du daemon (Running)

- `Storage Driver: overlay2` est le driver recommandé sur Linux — `vfs` indique un problème de configuration
- `Cgroup Version: 2` est la version moderne ; `1` est progressivement abandonnée
- `Docker Root Dir` : surveiller l'espace disque disponible sur cette partition
- `Warnings:` en bas de la sortie signale des problèmes de configuration à corriger

Sur Linux, si vous obtenez `permission denied`, ajoutez votre utilisateur au groupe docker :

```{code-cell} bash
sudo usermod -aG docker $USER
newgrp docker
```

### 02. Prise en main de Docker Desktop

::::{exercise}
:label: docker_ex02b
:nonumber:

**Durée estimée : 20 min** — Windows / macOS uniquement

Docker Desktop fournit une interface graphique pour piloter Docker sans ligne de commande. L'objectif est d'identifier les écrans principaux et de les mettre en correspondance avec les commandes CLI équivalentes.

**Étape 1 — Ouvrir Docker Desktop**

Lancer Docker Desktop depuis le menu démarrer (Windows) ou le dossier Applications (macOS). Vérifier que l'icône de la barre des tâches indique « Engine running ».

**Étape 2 — Onglet Containers**

Naviguer vers l'onglet **Containers**.

- Observer la liste des conteneurs (actifs et arrêtés)
- Identifier les colonnes : nom, image, statut, port, heure de création
- Lancer un conteneur depuis le terminal : `docker run -d --name desktop-test -p 8080:80 nginx:alpine`
- Vérifier qu'il apparaît dans la liste
- Cliquer sur le conteneur pour accéder à ses détails : Logs, Inspect, Stats, Terminal
- Ouvrir l'onglet **Logs** et comparer avec `docker logs desktop-test`
- Ouvrir l'onglet **Stats** et comparer avec `docker stats desktop-test --no-stream`
- Ouvrir l'onglet **Terminal** et comparer avec `docker exec -it desktop-test sh`

**Étape 3 — Onglet Images**

Naviguer vers l'onglet **Images**.

- Identifier les images présentes, leur taille et leur tag
- Comparer avec `docker images`
- Cliquer sur `nginx:alpine` : observer les couches et les métadonnées
- Utiliser le bouton **Pull** pour télécharger `alpine:latest` depuis Docker Hub

**Étape 4 — Onglet Volumes**

Naviguer vers l'onglet **Volumes**.

- Créer un volume : `docker volume create test-desktop`
- Vérifier qu'il apparaît dans la liste
- Comparer avec `docker volume ls`

**Étape 5 — Paramètres**

Ouvrir **Settings** (⚙️).

- **Resources** : noter les limites CPU, mémoire et swap allouées à Docker
- **Docker Engine** : visualiser la configuration JSON du daemon (`daemon.json`)
- **Proxies** : vérifier la configuration réseau

**Étape 6 — Nettoyer depuis Docker Desktop**

Dans l'onglet **Containers** :

- Sélectionner `desktop-test`
- Cliquer sur l'icône ⏹ (Stop) puis 🗑 (Delete)

Dans l'onglet **Volumes** :

- Sélectionner `test-desktop`
- Cliquer sur 🗑 (Delete)

Dans l'onglet **Images** :

- Sélectionner `nginx:alpine` et `alpine:latest`
- Cliquer sur 🗑 (Delete)

| Écran Docker Desktop | Commande CLI équivalente |
|----------------------|---------------------------|

### 04. Limiter et surveiller les ressources

::::{exercise}
:label: docker_ex03b
:nonumber:

**Durée estimée : 20 min**

Les **Cgroups** sont le mécanisme Linux que Docker utilise pour contraindre les ressources d'un conteneur. En production, un conteneur sans limite peut consommer toute la mémoire ou tout le CPU de l'hôte et déstabiliser les autres services.

**Étape 1 — Lancer un conteneur sans limite et observer**

```{code-cell} bash
docker run -d --name nginx-libre nginx:alpine
docker stats nginx-libre --no-stream
docker inspect nginx-libre | grep -E '"Memory"|"NanoCpus"'
```

Les valeurs `0` confirment l'absence de limite.

**Étape 2 — Lancer un conteneur avec des limites explicites**

```{code-cell} bash
docker run -d \
  --name nginx-limite \
  --memory 128m \
  --memory-swap 128m \
  --cpus 0.5 \
  nginx:alpine
```

Vérifier que les limites sont appliquées :

```{code-cell} bash
docker inspect nginx-limite | grep -E '"Memory"|"NanoCpus"'
docker stats nginx-limite --no-stream
```

**Étape 3 — Provoquer un dépassement mémoire**

```{code-cell} bash
docker run --rm --memory 64m alpine \
  sh -c "cat /dev/zero | head -c 128m > /dev/null"
```

Observer le comportement : le conteneur est tué par le kernel (OOM Killer) dès que la limite est dépassée.

**Étape 4 — Modifier les limites d'un conteneur en cours d'exécution**

```{code-cell} bash
docker update --memory 256m --cpus 1.0 nginx-limite
docker inspect nginx-limite | grep -E '"Memory"|"NanoCpus"'
```

**Étape 5 — Nettoyer**

```{code-cell} bash
docker stop nginx-libre nginx-limite
docker rm nginx-libre nginx-limite
docker image prune -f
docker volume prune -f
```

::::

::::{solution} docker_ex03b
:label: sol_docker_ex03b
:class: dropdown

**Étape 1 — `docker inspect` sans limite :**

```json
"Memory": 0,
"NanoCpus": 0,
```

`0` = pas de limite. Le conteneur peut utiliser toutes les ressources de l'hôte.

**Étape 2 — `docker inspect` avec limites :**

```json
"Memory": 134217728,
"NanoCpus": 500000000,
```

- `134217728` octets = 128 Mo
- `500000000` nanoCPUs = 0,5 CPU

**Étape 3 — Dépassement mémoire :**

```
Killed
```

Le processus est terminé par le kernel OOM Killer. `docker inspect` affichera `"OOMKilled": true` dans l'état du conteneur.

**Étape 4 — `docker update` :**

Les limites sont modifiables à chaud sans redémarrer le conteneur — utile pour absorber un pic de charge imprévu sans interruption de service.

**Quand utiliser les limites de ressources :**

- Toujours en production sur un hôte multi-conteneurs
- Quand plusieurs services partagent le même serveur (base de données, web, cache)
- Pour protéger l'hôte contre un conteneur défaillant ou malveillant
- En CI/CD pour éviter qu'un build runaway bloque les autres jobs

### 05. Isolation des processus et diagnostic

::::{exercise}
:label: docker_ex03c
:nonumber:

**Durée estimée : 20 min** — Linux uniquement

Les **Namespaces** donnent à chaque conteneur sa propre vue du système. Comprendre cette isolation est utile au quotidien pour le diagnostic : un conteneur a son propre PID 1, son propre espace réseau, son propre hostname.

**Étape 1 — Observer l'isolation des PID**

```{code-cell} bash
# Supprimer le conteneur s'il existe déjà
docker rm -f svc-nginx 2>/dev/null || true

docker run -d --name svc-nginx nginx:latest

# PID vu depuis l'hôte
ps aux | grep nginx

# PID vu depuis l'intérieur du conteneur via docker top (pas de ps requis)
docker top svc-nginx

# Alternative : entrer dans le conteneur et lire /proc directement
docker exec svc-nginx cat /proc/1/status | grep -E 'Name|Pid'
```

Le même processus a deux PID différents : un PID élevé sur l'hôte, PID 1 dans le conteneur.

**Étape 2 — Observer l'isolation du hostname**

```{code-cell} bash
hostname
docker exec svc-nginx hostname
```

**Étape 3 — Observer l'isolation réseau**

```{code-cell} bash
ip addr show
docker exec svc-nginx ip addr show 2>/dev/null || \
  docker exec svc-nginx cat /proc/net/fib_trie | head -20
```

**Étape 4 — Diagnostiquer un conteneur sans shell avec `nsenter`** (Linux uniquement)

Cas concret : certaines images de production ne contiennent pas de shell (`nginx:alpine`, distroless, scratch). `nsenter` permet d'entrer dans les namespaces du conteneur directement depuis l'hôte, sans passer par le daemon.

:::{note}
`nsenter` n'est pas disponible sur macOS ni Windows. Sur ces systèmes, utiliser `docker exec` avec une image qui embarque un shell, ou passer par `docker debug` (Docker Desktop ≥ 4.27).

```{code-cell} bash
# Alternative multiplateforme via docker debug
docker debug svc-nginx
```
:::

```{code-cell} bash
PID=$(docker inspect --format '{{.State.Pid}}' svc-nginx)
echo "PID hôte : $PID"

# Inspecter le réseau depuis le namespace du conteneur
sudo nsenter -t $PID --net -- ip addr show

# Inspecter les processus depuis le namespace du conteneur
sudo nsenter -t $PID --pid --mount -- ps aux
```

**Étape 5 — Nettoyer**

```{code-cell} bash
docker stop svc-nginx && docker rm svc-nginx
docker image prune -f
docker volume prune -f
```

::::{solution} docker_ex03c
:label: sol_docker_ex03c
:class: dropdown

**Étape 1 — PID depuis l'hôte :**

```
root  3842  nginx: master process nginx -g daemon off;
nginx 3871  nginx: worker process
```

**PID depuis `docker top` :**

```
UID    PID    PPID   C  STIME  TTY  TIME      CMD
root   3842   3821   0  10:00  ?    00:00:00  nginx: master process nginx -g daemon off;
nginx  3871   3842   0  10:00  ?    00:00:00  nginx: worker process
```

**PID via `/proc/1/status` depuis l'intérieur :**

```
Name:   nginx
Pid:    1
```

Depuis le conteneur, nginx est le PID **1**. Depuis l'hôte (`ps` ou `docker top`), le même processus a un PID élevé (ex. 3842). C'est l'effet du **Namespace PID**.

:::{note}
`docker top` interroge le daemon Docker qui lit `/proc` sur l'hôte — il ne nécessite aucun outil installé dans le conteneur. `cat /proc/1/status` fonctionne même dans les images les plus minimales (distroless, scratch) car `/proc` est un système de fichiers virtuel du noyau, toujours présent.
:::

**Étape 2 — hostname :**

```
# Hôte : mon-serveur
# Conteneur : a1b2c3d4e5f6  (ID court du conteneur)
```

Personnalisable avec `--hostname MON_NOM` au lancement.

**Étape 3 — interfaces réseau :**

```
# Hôte : eth0, docker0 172.17.0.1, ...
# Conteneur : lo, eth0@if19 172.17.0.2/16
```

Chaque conteneur a sa propre pile réseau via le **Namespace net**.

**Étape 4 — `nsenter` :**

```
# ip addr depuis le namespace réseau du conteneur
1: lo: <LOOPBACK,UP>
18: eth0@if19: <BROADCAST,UP> inet 172.17.0.2/16
```

**Quand utiliser `nsenter` :**

- Conteneur basé sur une image distroless ou scratch (pas de shell)
- Conteneur bloqué qui ne répond plus au daemon Docker
- Inspection réseau sans installer d'outils dans l'image de production
- Audit de sécurité depuis l'hôte sans modifier le conteneur

:::{note}
Cgroups et Namespaces ensemble constituent l'isolation complète d'un conteneur. Docker ne réinvente pas l'isolation — il orchestre ces primitives Linux pour les rendre accessibles et portables.
:::

## Comparer VM et conteneurs

### 06. Empreinte mémoire et temps de démarrage

::::{exercise}
:label: docker_ex00a
:nonumber:

**Durée estimée : 30 min**

**Étape 1 — Lancer un conteneur et mesurer la mémoire consommée**

```{code-cell} bash
docker run -d --name nginx-demo nginx:alpine
docker stats nginx-demo --no-stream
```

**Étape 2 — Mesurer le temps de démarrage**

```{code-cell} bash
time docker run --rm alpine echo "Conteneur démarré"
```

**Étape 3 — Inspecter la configuration mémoire/CPU allouée**

```{code-cell} bash
docker inspect nginx-demo | grep -E '"Memory"|"NanoCpus"|"CpuShares"'
```

**Étape 4 — Taille de l'image et des couches**

```{code-cell} bash
docker image inspect nginx:alpine | grep -E '"Size"|"VirtualSize"'
docker history nginx:alpine
```

**Étape 5 — Taille de la couche d'écriture du conteneur actif**

```{code-cell} bash
docker ps --size
```

**Étape 6 — Espace disque global occupé par Docker**

```{code-cell} bash
docker system df
```

Comparer avec l'empreinte d'une image VM Ubuntu : fichier `.vmdk` ou `.qcow2` de 4 à 20 Go sur le disque.

**Étape 7 — Lecture mémoire directement depuis le kernel** (Linux uniquement)

```{code-cell} bash
# Récupérer le PID du processus nginx sur l'hôte
NGINX_PID=$(docker inspect --format '{{.State.Pid}}' nginx-demo)
echo "PID hôte : $NGINX_PID"

# Lire la mémoire résidente depuis /proc
grep VmRSS /proc/$NGINX_PID/status
```

**Étape 8 — Entrer dans le namespace du conteneur sans docker exec** (Linux uniquement)

```{code-cell} bash
# Installer nsenter si absent
sudo apt-get install -y util-linux

# Entrer dans les namespaces PID et réseau du conteneur
sudo nsenter -t $NGINX_PID --pid --net -- ps aux
sudo nsenter -t $NGINX_PID --net -- ip addr show
```

**Étape 9 — Lister tous les namespaces associés au conteneur**

```{code-cell} bash
sudo ls -la /proc/$NGINX_PID/ns/
```

**Nettoyer**

```{code-cell} bash
docker stop nginx-demo && docker rm nginx-demo
docker image prune -f
docker volume prune -f
```

```
lrwxrwxrwx 1 root root 0 uts    -> uts:[4026532344]
```

Chaque namespace correspond à une dimension d'isolation : `pid` (processus), `net` (réseau), `mnt` (montages), `ipc` (IPC), `uts` (hostname), `user` (utilisateurs). Les numéros d'inode diffèrent de ceux de l'hôte — preuve de l'isolation effective.

| | Machine virtuelle | Conteneur Docker |
|--|-------------------|------------------|
| Empreinte mémoire | 256 Mo – 2 Go | 3 – 50 Mo |
| Taille image disque | 2 – 20 Go | 5 – 200 Mo |
| Temps de démarrage | 20 – 60 s | < 1 s |
| Isolation | Hyperviseur + OS invité | Namespaces + Cgroups |
| Visibilité depuis l'hôte | Opaque (hyperviseur) | Processus visible dans `/proc` |
| Densité sur un hôte | 10 – 20 VM | 100+ conteneurs |

## Exercice en autonomie - Choisir et configurer les images pour un site WordPress

::::{exercise}
:label: docker_ex_auto_01
:nonumber:

**Durée estimée : 30 min**

L'objectif est de constituer la liste des images nécessaires au déploiement d'un site ``WordPress`` en load balancing derrière un ``Nginx``, et de préparer la configuration sécurisée avant tout lancement.

Ce travail de sélection et de préparation précède systématiquement tout déploiement en production. Il n'y a pas de commande `docker run` à exécuter dans cet exercice : le résultat attendu est un répertoire de travail avec un fichier `.env` correctement rempli et un `README.md` justifiant les choix d'images.

**Étape 1** — Sur [hub.docker.com](https://hub.docker.com), rechercher les images officielles pour les trois composants de la pile : **WordPress**, **MariaDB** et **Nginx**. Pour chaque image, identifier les tags disponibles, distinguer les versions stables des versions de développement, et noter les variantes Alpine vs Debian.

**Étape 2** — Pour chaque image retenue, consulter la page Docker Hub et identifier les variables d'environnement supportées. Recenser au minimum : les variables obligatoires (sans lesquelles le conteneur ne démarre pas), les variables optionnelles utiles pour la production, et les ports exposés par défaut.

**Étape 3** — Créer un répertoire `wp-projet` et y créer un fichier `.env` rassemblant toutes les variables nécessaires : mots de passe MariaDB (root et utilisateur applicatif), nom de la base, nom d'utilisateur. Choisir des valeurs réalistes (pas `password` ni `123456`).

**Étape 4** — Ajouter un fichier `.gitignore` dans le répertoire pour exclure `.env` du contrôle de version. Vérifier que la règle est active.

**Étape 5** — Rédiger un court `README.md` (une douzaine de lignes) justifiant les choix : version de chaque image, raison du choix de la variante (Alpine ou Debian), liste des variables configurées et leur rôle.

**Étape 6** — Rédiger un court `CHANGELOG.md` (une douzaine de lignes) décrivant les modifications apportées : version de chaque image utilisée, justification du choix entre les variantes Alpine ou Debian, liste des variables d'environnement configurées et leur impact sur le fonctionnement de l'image.

**Critères de validation**

- Le répertoire `wp-projet` contient `.env`, `.gitignore` et `README.md`
- `.env` contient au moins : `MYSQL_ROOT_PASSWORD`, `MYSQL_DATABASE`, `MYSQL_USER`, `MYSQL_PASSWORD`
- `.gitignore` liste `.env`
- `README.md` justifie le choix de version pour chaque image
- Aucun mot de passe trivial (`password`, `123456`, `root`) dans `.env`

::::{solution} docker_ex_auto_01
:label: sol_docker_ex_auto_01
:class: dropdown

**Images recommandées pour la pile WordPress / load balancing**

| Composant | Image | Tag recommandé | Variante | Raison |
|-----------|-------|----------------|----------|---------|
| Base de données | `mariadb` | `10.11` | Debian | Version LTS, compatibilité garantie avec WordPress |
| Application | `wordpress` | `6.5-php8.3-apache` | Apache | Plus simple à démarrer ; intègre Apache |
| Reverse proxy / LB | `nginx` | `1.27-alpine` | Alpine | Image minimale ~5 Mo, surface d'attaque réduite |

**Variables d'environnement de l'image `mariadb` :**

| Variable | Obligatoire | Rôle |
|----------|-------------|-------|
| `MYSQL_ROOT_PASSWORD` | Oui | Mot de passe administrateur |
| `MYSQL_DATABASE` | Non | Crée une base au démarrage |
| `MYSQL_USER` | Non | Crée un utilisateur applicatif |
| `MYSQL_PASSWORD` | Non (requis si `MYSQL_USER`) | Mot de passe de l'utilisateur |

**Variables d'environnement de l'image `wordpress` :**

| Variable | Rôle |
|----------|-------|
| `WORDPRESS_DB_HOST` | Nom ou IP du serveur MariaDB |
| `WORDPRESS_DB_NAME` | Nom de la base de données |
| `WORDPRESS_DB_USER` | Utilisateur applicatif |
| `WORDPRESS_DB_PASSWORD` | Mot de passe |
| `WORDPRESS_TABLE_PREFIX` | Préfixe des tables (optionnel, sécurité) |

**Exemple de fichier `.env` :**

```{code-block} ini
:caption: Exemple de fichier .env
:filename: .env

# Base de données
MYSQL_ROOT_PASSWORD=R00t_S3cur3!2026   
MYSQL_DATABASE=wordpress_prod
MYSQL_USER=wp_app
MYSQL_PASSWORD=Wp_Appl1c@tion!

# WordPress
WORDPRESS_TABLE_PREFIX=wpx_
```

**Fichier `.gitignore` :**
```
.env
*.env
*.env.*
```

**Règles de sécurité pour les fichiers `.env` :**

- Ne jamais committer un `.env` contenant des secrets dans Git, même dans un dépôt privé
- Un secret poussé par erreur reste dans l'historique même après suppression — il faut le révoquer
- Utiliser des mots de passe d'au moins 16 caractères avec majuscules, chiffres et caractères spéciaux
- Ne pas utiliser le même mot de passe pour `MYSQL_ROOT_PASSWORD` et `MYSQL_PASSWORD`
- En production, préférer un gestionnaire de secrets (Vault, Docker Secrets) plutôt qu'un fichier `.env`
- Vérifier les permissions : `chmod 600 .env` (lecture propriétaire uniquement)

**Pourquoi Alpine pour Nginx et Debian pour MariaDB / WordPress ?**

Nginx en Alpine (~5 Mo) : surface d'attaque minimale, temps de pull rapide en CI/CD, parfaitement suffisant pour un reverse proxy sans extensions système.

MariaDB et WordPress en Debian : les extensions PHP et les drivers MySQL dépendent de bibliothèques système absentes d'Alpine ; la compatibilité est plus large et les images officielles sont mieux testées sur cette base.
::::
